# 复盘与方案：手势鼠标的点击稳定性与指针锁定

> 日期：2026-06-13  ·  范围：仅方案，不改代码
> 主题：如何让"手势代替鼠标"稳定、可控、实用，重点解决**点击瞬间指针漂移**
> 所有结论均基于对 main 当前代码的实证走读（附 `file:line`）

---

## 0. TL;DR（结论先行）

1. **你的直觉是对的**：系统无法自己可靠判断"我已经到目标了"，由左手显式告诉它，是最可控的方案。
2. **但根因不在"左手要不要锁"，而在结构**：指针控制点（食指尖 lm8）和点击触发（拇指→食指捏合）**共用同一根手指**，所以捏合一定扰动指针。这在单手模式和双手模式里**都存在**。
3. **最高杠杆的修复不是左手锁，而是"把捏合状态耦合进指针引擎"**——目前两者完全解耦 [app.py:569-716]，指针引擎根本不知道你在捏合。一旦耦合，可在"捏合意图出现的那一刻"冻结指针，对**所有模式**都生效，且改动很小。
4. **左手拳锁定仍然值得做**，但定位为**第二层的精确模式**，而不是唯一手段；并且必须先解决它和"右键 = 左拳"的语义冲突。
5. **推荐分层落地**：P0 捏合自动冻结（自动、通用、小改） → P1 左拳显式锁定（精确、可控） → P2 指针控制点可选（结构性根治） → P3 GUI 暴露参数。

## 1. 根因分析：为什么点击一定漂移

### 1.1 指针是相对位移驱动
`PointerEngine.update()` 每帧计算归一化食指尖的**帧间位移** `u = 本帧point − 上帧point`，再乘 `速度增益 × 深度因子 × pixel_per_palm_width`，输出整数 `ScreenDelta`，由 `mouse.move_relative()` 经 `MOUSEEVENTF_MOVE` 下发 [pointer_engine.py:112-131; mouse_controller.py:86-91]。
→ 这是"鼠标式相对控制"，不是触摸板式绝对映射（Phase4 研究也明确推荐相对控制 [phase4-pointer-control-paradigms.md:3-12,341-360]）。

### 1.2 控制点与点击触发共用食指
- 指针运动源：食指尖 **lm8** [pointer_engine.py:69]
- 左键捏合：拇指 **lm4** ↔ 食指 **lm8** 的距离 [gesture_detector.py:62-67]

捏合 = 让 lm4 靠近 lm8，这个动作**必然移动 lm8**。因为指针吃的是 lm8 的帧间位移，所以点击动作直接被翻译成指针漂移。**这是结构性的，不是操作问题。**

### 1.3 指针引擎与手势引擎完全解耦
`_run_loop` 里 `pointer.update()` 和 `gesture.update()` 在同一 try 块里**先后独立**执行 [app.py:569-716]。`PointerEngine` 对 pinch 状态一无所知，没有任何"点击时冻结/锚定"逻辑。两者唯一的偶合是都读同一个 pointer hand 观测。
→ **修复的抓手就在这里**：只要让指针引擎感知到"正在捏合"，就能在点击窗口内停掉位移。

### 1.4 现有的抖动缓解（都不是 click-aware）
| 机制 | 作用 | 局限 |
|---|---|---|
| 速度死区 gain=0 (速度 < v_jitter=0.20) [pointer_engine.py:239-244] | 手不动时不漂 | 一次**有意**的快速捏合速度轻松超过 0.20，照漂 |
| 4 帧速度均值选增益 [pointer_engine.py:36-59] | 平滑的是"增益选择"，**不是位置** | 位置本身逐帧裸传 |
| 亚像素余量累计 _rx/_ry [pointer_engine.py:133-141] | 不丢分像素，减少取整抖 | 与点击无关 |
| `dead_zone_radius` | 位移死区 | **默认 0.0 = 关闭** |

### 1.5 关键缺口：没有位置平滑
Phase4 研究**明确建议**位置一阶 EMA（alpha 0.2–0.35）+ 可选 3–5 帧中值滤波 [phase4-pointer-control-paradigms.md:397-401]，但**配置里没有任何平滑字段，引擎里也没有实现**。当前全靠速度死区。这是研究文档与实现之间的一处明显落差，独立于点击漂移也值得补。

## 2. 当前双手交互模型（实证）

```
左手（mode 手，dominant_hand=right 时为 Left）：
  张开手掌  → ARM/保持 gate ACTIVE（移动模式）
  握拳      → mode_is_secondary=True = 右键修饰   ← 冲突点
右手（pointer 手）：
  食指尖移动 → 指针移动
  拇指-食指捏合 release → 左键（300ms 内连发 → 双击）
  捏合 + 移动 > 0.35*palm → 拖拽
双手模式下：thumb-middle 右键路径、alt_tab、grab_scroll、shortcut 全部禁用
```
依据：[bimanual_gate.py:65-72,80-150; gesture_detector.py:120-158,160-201; app.py:526-621]

**两个要点：**
- 左拳今天**只翻转右键，不冻结指针**——gate 仍 ACTIVE，指针照动 [bimanual_gate.py:124-132]。所以"左拳锁定"是新语义，不是改个开关。
- `mode_is_secondary` 一个布尔扛了两个含义的潜在冲突：现在是"右键"，你想让它变"锁定"。**不能简单覆盖**，否则"左拳+右手捏合"到底是锁定后左键、还是右键，无法区分。

## 3. 直接回答你的两个疑问

> **Q1：右手要 pinch 就必须动手指，靠"检测到 pinch_close 后冻结"是不是太晚？**

是的，太晚。`pinch_close_ratio=0.5` 且点击 emit-on-release [gesture_detector.py:14-22,120]，等确认到 close 时食指早已偏移。**所以不能用"确认捏合后冻结"，要用"捏合意图一出现就冻结"**——见 P0：在比 0.5 更早的"预关闭"阈值（如 0.7，刚好是 open 阈值）上触发冻结，相当于"拇指刚开始靠近就停指针"。

> **Q2：如果靠延迟，怎么知道鼠标已经到我要的位置？**

不靠延迟判断"到没到"。换个定义：**"目标位置" = 你开始捏合之前、指针最后一次静止的位置。** 因为人在点击前天然会先停在目标上。做法：维护一个最近 ~150–200ms 的位置环形缓冲，每帧记录位置+当时指尖速度；捏合意图一出现，就把点击锚点取为"缓冲里最后一个速度低于静止阈值的位置"。这样系统不需要"猜你到没到"，它直接复用"你刚才停下的那一点"。这也正是为什么 P0（自动）能成立，而不必依赖左手。

**但如果你点击前不停顿（边移边捏）**，缓冲里没有静止点，P0 退化为"当前位置冻结"，仍可能小漂——这种精确场景正是 P1 左手锁定的用武之地。两层互补。

## 4. 方案空间（三角度）

### 角度 A — 显式左手锁定（你的想法，精炼版）
左手升级为真正的模式选择器：张开=移动、握拳=锁定（冻结指针，右手照常点击/拖拽）。
- 优点：直接回答"到没到"——由你显式发信号；确定、可控；完全消除漂移（锁定时根本不调 `move_relative`）。
- 缺点：每次点击都要双手协调（疲劳、变慢）；和"右键=左拳"冲突，必须给右键搬家。

### 角度 B — 无需额外动作的自动稳定
- **B1 捏合意图冻结 + 锚点回退**（见 Q2）：环形缓冲 + 预关闭阈值，自动、通用。
- **B2 指针控制点与捏合解耦**：指针改用**手掌中心**（lm 0,5,9,13,17 均值，`PoseDetector` 已在算 [pose_detector.py:9-44]）或食指 MCP(lm5)，则拇指-食指捏合几乎不动控制点——**结构性根治**。代价：手掌中心指点手感不同（精细控制变弱）。
- **B3 速度门控驻留冻结**：指尖速度连续 K 帧低于阈值 → 软锁，只有大动作能突破。启发式"到位检测"。风险：慢速跟踪时误锁。
- **B4 驻留点击（dwell-to-click）**：悬停 T ms 触发点击，彻底不用捏合。代价：慢、易误触（Midas touch），仅适合做无障碍可选项。

### 角度 C — 务实分层路线
把最便宜的通用赢（B1）放 P0，把你的显式锁（A）放 P1，把结构根治（B2）放 P2。下一节展开。

## 5. 推荐方案：分层落地

### P0 —— 捏合自动冻结（自动 / 通用 / 小改）⭐ 最高杠杆
**做什么**：把 `GestureDetector` 的捏合状态耦合进指针冻结。当拇指-食指比率跨过**预关闭阈值**（建议 ≈ `pinch_open_ratio` 0.7，远早于 close 0.5）那一刻，冻结指针，直到 release + cooldown 后解冻并重新锚定。

**为什么能成立 / 回答 Q1Q2**：在拇指"刚开始靠近"时就冻结，绝大部分捏合位移发生在冻结之后；锚点取自环形缓冲里"最后一次静止位置"，不依赖延迟判断到位。

**改哪里**：
- 跨帧顺序：用**上一帧**的 pinch 状态门控**本帧**的 `pointer.update`，或在 `_run_loop` 内先取捏合比率再决定是否下发 `move_relative` [app.py:569-716]。
- 冻结实现两档：
  - **MVP（最小）**：冻结期间强制 `gain=0`（或 app 跳过 `move_relative`）。利用现有 `reset()`/`_anchor()`——解冻首帧只锚定不移动，天然不跳变 [pointer_engine.py:108-110,159-172]。无需回退，"静止→预关闭"间的残余漂移由"预关闭阈值设早"来压到几像素。
  - **进阶（回退到锚点）**：因为是相对位移，回退需在小窗口内累计已下发的 delta，冻结时反向发一个 delta 把指针拉回缓冲锚点。
- 复用现有 `last_velocity` 只读属性做静止判定 [pointer_engine.py:174-192]。
- 失效保护：若预关闭后 N 帧内没真正形成点击，自动解冻（避免卡死）。

**收益**：单手模式 + 双手模式同时受益；不需要用户养成新习惯。
**工作量**：S。**风险**：预关闭阈值设太早会"想微调却被冻住"——需可调 + hysteresis。

### P1 —— 左拳显式锁定（精确 / 可控）= 你的需求
**做什么**：左手张开=移动，左手握拳=锁定（指针冻结，右手可点击/右键，不漂）。作为 P0 之上的**精确模式**，处理"小按钮、不停顿直接点"的场景。

**关键前置——拆分被重载的 `mode_is_secondary`**：
把 `BimanualGateResult` 由两标志扩成三标志：
```
gate_active     : 是否允许双手控制（同今天）
pointer_frozen  : 左拳 → True（新；冻结指针）
secondary_click : 右键修饰（从左拳搬走，见下）
```
- `app.py` 双手分支：`bimanual_pointer_active = ... AND not pointer_frozen`；冻结时调 `pointer.reset()` 并跳过 `move_relative` [app.py:559-621]。
- 解冻（拳→掌）首帧由 `_anchor()` 重新定origin，无跳变 [pointer_engine.py:108-110]。

**右键搬家（这是 P1 的真正阻塞，必须先定）**——三选一：
| 方案 | 做法 | 评价 |
|---|---|---|
| **(b) 推荐** | 在双手模式**重新启用右手 thumb-middle 捏合 = 右键**（该路径已实现且有测试，只是双手模式被禁 [gesture_detector.py:160-201]） | 复用已测代码，**完全释放左手**；左手语义干净=只管"移动/锁定"；thumb-middle 的漂移由 P0 覆盖 |
| (a) | 左手 V 字（食指+中指）= 右键修饰，需新增 `PoseDetector.is_v_sign` | 语义清晰，但要新姿态分类器 + 识别鲁棒性风险 |
| (c) | 锁定态下右手另一手势 = 右键 | 右手负担重，不如 (b) |

→ **建议 (b)**：左手 = {张开移动 / 握拳锁定}，右键交还给右手 thumb-middle。这样左拳冲突彻底消失，且最少新代码。

**最终 v2 模型**：
```
左手张开 → 移动模式（指针跟随右手食指）
左手握拳 → 锁定模式（指针冻结）
右手 拇指-食指 捏合 → 左键（建议在锁定态点小目标）
右手 拇指-中指 捏合 → 右键（重新启用）
拖拽 → 移动模式下 捏合+移动（锁定态与拖拽语义矛盾，拖拽留在移动模式）
```
**工作量**：M。**风险**：双手协调疲劳——所以 P0 先行，让"不锁也能点"，左拳只在精确场景才用。

### P2 —— 指针控制点可选（结构性根治）
**做什么**：配置项 `pointer_landmark = index_tip | palm_center`。选 palm_center 时，捏合（动拇指/食指）几乎不动控制点，从根上消除耦合。
**改哪里**：`PointerEngine` 取点处 [pointer_engine.py:69]；`PoseDetector` 已有手掌中心算法可复用 [pose_detector.py:9-44]。
**工作量**：S–M。**风险**：手感变化（指尖→掌心，精细控制变弱），需用户试用对比，故设为可选而非默认。

### P3 —— GUI / 配置暴露
P0/P1 的新参数（预关闭阈值、静止速度阈值、冻结失效超时、`pointer_frozen` 启用、`pointer_landmark`）接入 `advanced_gui.py` 与 `BimanualConfig`/schema，与现有双手参数同一卡片。注意 `pointer_landmark` 若运行时切换可热更，`enabled` 类才需重启。
**工作量**：S。

### （独立项）补位置平滑
对齐 Phase4 研究 [phase4:397-401]：在 `PointerEngine` 加一个可配 one-euro / 一阶 EMA（alpha 0.2–0.35），降低全局抖动。与点击漂移正交，但显著提升整体"稳定可控"观感。建议在 P0 之后排入。

## 7. 风险与测试要点（落地时）

- **解冻跳变**：必须验证 拳→掌 / release→解冻 首帧只锚定不跳（已有 `_anchor` 保证，需测试覆盖）[pointer_engine.py:108-110]。
- **误冻结**：预关闭阈值过早会让"想微调"被冻住——参数化 + hysteresis + 失效超时。
- **快速点击退化**：边移边捏无静止锚点时 P0 退化为当前位置冻结；用 P1 兜底，测试两条路径。
- **右键回归**：重新启用 thumb-middle 后，确认它在双手模式与左键互斥、不误触（参考 [gesture_detector.py:89-95,163-167] 的互斥逻辑）。
- **安全门不回退**：P0/P1 改动不得绕过 `gate_active` / `quality.ok` / `is_active` 的现有安全门 [app.py:559-621]。
- **CLAUDE.md 要求**：改任一符号前先跑 `impact()`，提交前 `detect_changes({scope:"compare", base_ref:"main"})`；`PointerEngine.update`、`BimanualGate.update`、`GestureDetector.update` 都是高扇出热点，预计 HIGH 风险，动前务必看 blast radius。

## 8. 一句话给你

先做 **P0 捏合自动冻结**（小改、单双手通用、直接答复"太晚/怎么判断到位"两个疑问），它能消掉大部分漂移；再上 **P1 左拳锁定**作为精确模式，同时把右键从左拳搬到右手 thumb-middle，干净解决语义冲突。你的方案是对的，只是它最适合当"第二层精确手段"，而不是唯一手段。
